# Отчет о КР1 по КПО - HSE-BANK

## 1. Общая идея решения

В данном решении реализовано консольное приложение **HSE-BANK**, который позволяет:
- **Управлять финансовыми данными:**
  - **BankAccount:** Создание, редактирование и удаление счетов.
  - **Category:** Создание, редактирование и удаление категорий (доходов и расходов).
  - **Operation:** Регистрация операций (доходов и расходов) с автоматическим обновлением баланса счета.
- **Аналитика:**
  - Подсчёт чистой разницы между доходами и расходами за указанный период.
  - Группировка операций по категориям.
- **Импорт и экспорт данных:**
  - Реализована возможность импорта и экспорта данных в форматы **JSON**, **CSV** и **YAML**. Экспорт данных производится в файлы с корректным форматированием (например, JSON с отступами и правильными разделителями).
- **Консольное меню для взаимодействия с пользователем:**
  - Предоставлено интерактивное меню, через которое можно создавать счета, категории, операции, выполнять аналитику и экспорт данных.
- **Тестирование:**
  - Реализовано покрытие ключевых компонентов (например, фасада, репозитория, импортёров/экспортёров, меню) unit‑тестами с использованием NUnit.

## 2. Принципы SOLID и GRASP

### SOLID

- **Single Responsibility Principle (SRP):**  
  Каждый класс имеет одну зону ответственности.  
  - К примеру, классы `BankAccount`, `Category` и `Operation` отвечают только за хранение и изменение своего состояния. Класс `FinanceFacade` объединяет операции с различными сущностями.

- **Open/Closed Principle (OCP):**  
  Система открыта для расширения (новые форматы импорта/экспорта, новые команды) и закрыта для модификации существующего кода.  
  - Абстрактный класс `DataImporter` позволяет добавлять новые реализации для разных форматов без изменения базовой логики.

- **Liskov Substitution Principle (LSP):**  
  Реализации интерфейсов можно взаимозаменять без нарушения корректности работы системы.

- **Interface Segregation Principle (ISP):**  
  Интерфейсы разделены по функциональности, что уменьшает нагрузку на их реализации.

- **Dependency Inversion Principle (DIP):**  
  Консольное приложение зависит от абстракций, а не от конкретных реализаций.  
  - *Пример:* Внедрение зависимостей через DI-контейнер (Microsoft.Extensions.DependencyInjection) позволяет легко подменять реализации при тестировании или дальнейшем расширении функционала.

### GRASP

- **High Cohesion (Высокая связность):**  
  Каждый класс отвечает за строго определённую задачу.  
  - Допустим, класс `DataRepositoryProxy` занимается исключительно CRUD-операциями над сущностями.

- **Low Coupling (Низкая связанность):**  
  Компоненты взаимодействуют через четко определенные интерфейсы, что снижает зависимость между модулями.  
  - *Пример:* `FinanceFacade` использует интерфейс `IDataRepository`, что позволяет легко заменить реализацию хранилища данных.

## 3. Реализованные паттерны GoF

### Порождающие паттерны

- **Фабрика (Factory):**  
  - **Реализация:** Класс `DomainFactory`  
  - **Обоснование:** Обеспечивает централизованное создание доменных объектов (счетов, категорий, операций) с валидацией, что предотвращает дублирование кода и поддерживает инварианты.

### Структурные паттерны

- **Фасад (Facade):**  
  - **Реализация:** Класс `FinanceFacade`  
  - **Обоснование:** Предоставляет упрощённый интерфейс для взаимодействия с различными подсистемами (счета, категории, операции) и скрывает сложность внутренних процессов.

- **Прокси (Proxy):**  
  - **Реализация:** Класс `DataRepositoryProxy`  
  - **Обоснование:** Реализует in-memory репозиторий, позволяющий организовать кэширование данных и упрощает замену механизма хранения (например, для интеграции с БД).

### Поведенческие паттерны

- **Команда:**  
  - **Реализация:** Интерфейс `ICommand` и его реализации (например, `CreateAccountCommand`)  
  - **Обоснование:** Инкапсулирует операции пользователя (создание счетов, операций) и позволяет легко управлять выполнением сценариев, а также расширять функциональность.

- **Декоратор:**  
  - **Реализация:** Класс `CommandTimerDecorator`  
  - **Обоснование:** Оборачивает команды для измерения времени их выполнения, не вмешиваясь в основную логику, что позволяет добавить дополнительное поведение.

- **Шаблонный метод:**  
  - **Реализация:** Абстрактный класс `DataImporter` с конкретными реализациями (`JsonDataImporter`, `CsvDataImporter`, `YamlDataImporter`)  
  - **Обоснование:** Определяет скелет алгоритма импорта данных, позволяя подклассам реализовывать только отличающуюся часть (парсинг данных).

- **Посетитель:**  
  - **Реализация:** Интерфейс `IDataExportVisitor` и его реализации для экспорта в файлы (`FileJsonExportVisitor`, `FileCsvExportVisitor`, `FileYamlExportVisitor`)  
  - **Обоснование:** Разделяет логику экспорта данных от доменных объектов, позволяя легко добавлять новые форматы экспорта без модификации базовых классов.

## 3. Тестирование

- **Unit-тесты:**  
  Для проверки ключевых компонентов (фасада, репозитория, импортёров/экспортёров, меню) используются unit‑тесты, реализованные с помощью NUnit.  
  Запустить тесты можно через Test Explorer в Visual Studio или через IDE Rider от JetBrains.
